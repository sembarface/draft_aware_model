# Контекст проекта

Это курсовой ML-проект по Dota 2, посвященный поддержке принятия решений на стадии драфта. Основная идея - по текущему состоянию драфта оценивать доступных героев и ранжировать их отдельно для действий `pick` и `ban`.

Изначально проект был ближе к статистическому анализу матчей, героев и патчей. Сейчас он расширяется в полноценный ML-пайплайн: из raw-таблиц строятся события драфта, затем состояния драфта, затем candidate table для обучения ранжирующей модели.

## Постановка задачи

Задача формулируется через candidate table. Одна строка в `draft_candidates` - это один доступный герой-кандидат в конкретном `state_id`.

Для каждого состояния драфта строится набор строк по всем героям, которые еще доступны. Целевая переменная `target` равна `1` для героя, который был реально выбран или забанен в этом состоянии, и `0` для остальных кандидатов.

Отдельно строятся таблицы:

- `draft_candidates_pick`
- `draft_candidates_ban`

Такой формат позволяет обучать бинарную модель, а затем использовать ее score как основу для ранжирования кандидатов внутри каждого `state_id`.

## Текущий baseline

Первые baseline-модели обучены на CatBoost.

Метрики для pick:

- `Top-1` ≈ 0.119
- `Top-5` ≈ 0.346
- `Mean Rank` ≈ 18.26

Метрики для ban:

- `Top-1` ≈ 0.104
- `Top-5` ≈ 0.323
- `Mean Rank` ≈ 15.08

Главный вывод: модель работает и дает осмысленное ранжирование, но пока сильно зависит от популярности героев. Текущие признаки в основном описывают фазу драфта, сторону, команды и общую статистику кандидата.

## Следующие исследовательские шаги

Дальше стоит аккуратно добавлять признаки, которые лучше описывают контекст конкретного драфта:

- командные признаки;
- синергии между уже выбранными героями;
- контрпики против вражеских героев;
- `player x hero` признаки.

Эти признаки нужно добавлять без target leakage и без использования post-match информации.
